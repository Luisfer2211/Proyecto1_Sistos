CC3064 - Proyecto 01 Chat Multithread
Definición del protocolo de aplicación (borrador para documento PDF)
Transporte: TCP (sockets stream). Codificación: UTF-8. Fin de mensaje: '\n'.
================================================================================

1. Establecimiento de conexión y registro
   Tras accept(), el cliente envía exactamente una línea:
      REGISTER <nombre_usuario>
   Reglas del servidor (según enunciado):
   - Acepta solo si el nombre NO está en uso Y la IP de origen NO está ya
     registrada para otro cliente conectado.
   Respuestas del servidor (una línea):
      OK REGISTER
      ERR DUPLICATE_NAME
      ERR DUPLICATE_IP
      ERR INVALID_NAME     (opcional: nombre vacío / caracteres no permitidos)

2. Mensajes durante la sesión (cliente -> servidor)
   Todas son una línea; el servidor valida y responde o reenvía.

   LIST
      Solicita listado de usuarios conectados.
      Respuesta multilínea posible:
         OK LIST BEGIN
         <usuario1>
         <usuario2>
         ...
         OK LIST END

   WHOIS <nombre_usuario>
      Consulta IP asociada a un usuario conectado.
         OK WHOIS <nombre_usuario> <ip>
         ERR NOT_FOUND

   STATUS <ACTIVO|OCUPADO|INACTIVO>
      El cliente pide cambiar su estado. El servidor actualiza y confirma.
         OK STATUS <ACTIVO|OCUPADO|INACTIVO>
         ERR INVALID_STATUS

   (Chat) Formato acorde al enunciado del cliente: "<destino> <mensaje>"
   En el protocolo interno propuesto (lo envía el cliente ya parseado):
      CHAT ALL <texto>                    -> broadcast a todos
      CHAT <usuario_destino> <texto>      -> mensaje directo

   Servidor -> cliente(s) al entregar mensajes:
      IN CHAT FROM <origen> TO ALL <texto>
      IN CHAT FROM <origen> TO <destino> <texto>

   PING (opcional, para mantener sesión y medir inactividad)
      Cliente envía periódicamente o el servidor usa timeout de lectura.

   BYE
      Cierre ordenado. Servidor elimina al usuario de la lista y cierra el socket.
         OK BYE

3. Inactividad (status INACTIVO)
   - El servidor marca INACTIVO si no recibe tráfico del cliente en T segundos
     (T configurable, valor corto para demostración en clase).
   - Puede usar alarm/select con timeout o thread watchdog; quedará documentado
     en el informe.

4. Consideraciones de implementación
   - Un thread por cliente conectado (modelo del curso) + thread principal que
     hace accept().
   - Estructuras compartidas (lista de sesiones: nombre, IP, fd, estado, mutex)
     protegidas con pthread_mutex_t (justificación en el PDF).
   - Dos condiciones de carrera típicas a analizar en el documento:
     (a) Registro simultáneo de dos clientes con el mismo nombre.
     (b) Broadcast concurrente con desconexión / borrado de un usuario de la lista.
